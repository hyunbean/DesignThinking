> 2026-07-23 측정. 이 스킬이 실제로 스펙대로 동작하는지, 단계 격리 설계가 문서 수준에서

## 1. 정적 분석 — 에이전트 스펙 11개

`.claude/agents/dt-*.md` 11개 파일을 파싱해 스펙 준수 여부를 셌다.

| 항목 | 결과 |
|---|---|
| 에이전트 수 | 11개 (총 465줄) |
| 표준 5섹션(역할/배경/임무/출력양식/규칙) 완비 | **9 / 11** |
| 출력 양식 명시 | **9 / 11** |
| 출력 연결(다음 단계 명시) | **4 / 11** |
| `tools` 권한 선언 | **11 / 11** |

### 최소 권한 원칙이 실제로 적용되어 있음

| 권한 | 에이전트 | 개수 |
|---|---|---|
| `Read` 만 | 텍스트 산출 단계 전부 (E1·E2·D1·D2·D3·I1·P1·T1·A1) | 9 |
| `Read, Write, Bash` | 이미지 생성 단계 (I2·P2) | 2 |

파일을 쓰거나 스크립트를 실행할 필요가 있는 **두 단계에만** 쓰기·실행 권한이 부여되어 있고,
나머지 9개는 읽기 전용이다. 각 서브에이전트에 필요한 최소 권한만 주는 설계가 문서 수준에서
확인된다.

### 발견된 불일치 (개선 대상)

- `dt-i2-sketch`, `dt-p2-illustration` 두 이미지 단계만 표준 5섹션 구조를 따르지 않는다
  (출력 양식 섹션 없음). 이미지 프롬프트를 다루는 단계라 구조가 다르지만, 최소한
  출력 파일 규약은 같은 형식으로 명시하는 편이 일관적이다.
- "출력 연결"(이 단계의 산출물이 다음 어느 단계로 가는지)을 명시한 것은 11개 중 4개뿐이다.
  파이프라인 구조는 `SKILL.md`에 있지만, 에이전트 파일만 봐서는 전후 관계를 알 수 없는
  경우가 많다.

## 2. 실행 측정 — Define 체인 (D1 → D2 → D3)

파이프라인 전체(11단계) 중 사람 개입 구간([E2]→[D1] 실제 인터뷰)과 이미지 생성 단계를 제외한
**완전 자동 구간인 Define 3단계**를 실제로 실행했다.

**입력:** [`examples/2026-07-11-goal-pet-saving/interview-notes.md`](../examples/2026-07-11-goal-pet-saving/interview-notes.md)
(실제 캡스톤 인터뷰 1건, 익명 처리)
**PROJECT_TOPIC:** 대학생의 소비 관리와 저축 습관 형성을 돕는 금융 서비스 개발
**실행 방식:** 각 단계를 독립된 서브에이전트로 호출하고, 이전 단계 출력을 다음 단계 입력에
그대로 전달 (`SKILL.md` 데이터 전달 규칙 1)

### 소요 시간·비용

| 단계 | 소요 | 토큰 |
|---|---|---|
| [D1] 클러스터링 | 34.9s | 36,818 |
| [D2] POV 작성 | 47.8s | 38,053 |
| [D3] 우선순위 평가 | 70.2s | 40,271 |
| **합계** | **152.8s (2.5분)** | **115,142** |

### 형식 준수

각 단계 출력이 해당 스펙의 "출력 양식" 필수 요소를 갖췄는지 확인:

| 단계 | 필수 요소 | 결과 |
|---|---|---|
| D1 | `[클러스터링 결과]` / 카테고리 목록 / 카테고리별 클러스터링 | **3/3** |
| D2 | `[POV 진술서 목록]` / 사용자·욕구·통찰·문장 4필드 | **5/5** |
| D3 | `[POV 우선순위 평가]` / 점수표 / Ranked Priority / 영향력·차별화 축 | **5/5** |

### 단계 간 데이터 보존

`SKILL.md` 데이터 전달 규칙 1은 "이전 단계 출력을 임의로 요약하거나 문장을 바꾸지 않는다"고
정한다. 실제로 지켜졌는지 개수로 확인:

| 구간 | 결과 |
|---|---|
| D1 카테고리 8개 → D2 POV 8개 | 1:1 대응, 누락 0 |
| D2 POV 8개 → D3 평가 8개 | 전량 점수화, 누락 0 |
| D3 최종 선별 | 상위 5개 (스펙의 "상위 3~5개" 범위 내) |

카테고리 수(8)는 스펙이 정한 개수가 아니라 입력 데이터에서 도출된 값이며, 같은 입력으로
만든 [기존 예시](../examples/2026-07-11-goal-pet-saving/D1-clustering.md)에서는 5개가
나왔다. 즉 **클러스터 개수 자체는 실행마다 달라질 수 있고**, 스펙이 보장하는 것은 개수가
아니라 **형식과 단계 간 전달의 무결성**이다.

## 3. 한계

- Define 3단계만 측정했다. 전체 11단계, 특히 이미지 생성 단계([I2]·[P2])는 측정에서 제외했다.
- **"스킬 없이 수동으로 했을 때"와의 비교 기준선(baseline)이 없다.** 따라서 "N% 빨라졌다"
  같은 개선율은 주장할 수 없고, 이 문서는 "스펙대로 동작하는가"만 확인한다.
- 1회 실행 결과다. LLM은 완전히 결정론적이지 않으므로 소요 시간·클러스터 개수는 실행마다
  달라진다. 형식 준수 항목은 스펙 기반이라 상대적으로 안정적이다.

## 4. 재현 방법

```
PROJECT_TOPIC = 대학생의 소비 관리와 저축 습관 형성을 돕는 금융 서비스 개발
입력 = examples/2026-07-11-goal-pet-saving/interview-notes.md

[D1] .claude/agents/dt-d1-clustering.md 스펙으로 실행 → 클러스터링 결과
[D2] [D1] 출력을 그대로 입력으로 전달 → POV 목록
[D3] [D2] 출력을 그대로 입력으로 전달 → 우선순위 평가
```

정적 분석은 `.claude/agents/dt-*.md`의 frontmatter(`tools`)와 본문 섹션 헤더를 세면 재현된다.
